📋 Project Overview & Problem Statement

Challenge: A small online seller runs vouchers constantly and sales spike every campaign, but they cannot tell which vouchers drove genuinely new sales and which simply subsidised buyers who would have purchased anyway. A single flat voucher policy applied to every category quietly leaks margin across hundreds of SKUs.

Solution: Voucher Analyzer estimates true promotion incrementality, layers a real cost stack (COGS, platform fee, ad allocation), scores every voucher, and turns the flat policy into a per-category plan. It also proves its own estimates against a baked-in ground truth, so the numbers can be trusted rather than taken on faith.

Key Benefits

  • Cause vs coincidence: separates incremental sales from subsidised baseline, so raw campaign sales stop misleading.
  • Per-voucher verdict: a scorecard of net P&L, ROI and subsidy share with a keep, scale, or kill call.
  • Targeted, not blunt: a category by voucher margin matrix replaces one rule for every category.
  • Answers on the page: the ten questions sellers ask most, each answered in plain language with a chart.
  • Governed live queries: four read-only SQL drill-downs run in the browser, with the query on display.
  • Proven, not asserted: a recovery check grades the estimate against a hidden truth (about 45% incremental versus a true 46%, beating the naive 100% read).

🖥️ Application Features

🔎 Incrementality Audit

A baseline-counterfactual estimate of the sales each voucher truly caused, broken out by category, mechanic (fixed vs percentage), depth, and campaign tier.

📊 Voucher Scorecard

Net P&L, ROI, redemption rate and effective discount for all seven vouchers, each with a keep, scale, or kill verdict.

🧮 Category by Voucher Matrix

Margin contribution for every category and voucher pair, surfacing the categories that lose money on vouchers no matter what.

❓ The Key Questions

Ten governed lookups answered in plain language with an inline chart. Click a question to open the answer, click again to close.

⚡ Live Governed Queries

Four fixed, parameterised, read-only SQL tracks executed in the browser via sql.js, each showing the exact query that ran.

✅ Recovery Check

The estimate validated against a baked-in ground truth, with the naive gross-sales number shown alongside for contrast.

🤖 AI Integration & Intelligence

📈 Promotion Incrementality Estimation

A baseline-counterfactual / difference-in-differences method estimates the unobservable "would have bought anyway" share. This is the core analytical skill the project demonstrates.

🔐 Governed Query Layer

Built on the Klinik Dr Fang pattern: the assistant can only pick a fixed track and bind parameter values, never author SQL. Every query is read-only, keyless, and runs client-side.

🎯 Evaluation Discipline

Every estimate is graded against a known truth that is withheld from the deployed data, so the tool genuinely estimates the answer rather than reading it.

💰 Cost-Aware Scoring

A full cost stack (COGS, platform fee, ad allocation) turns raw redemption counts into the real margin each voucher added or destroyed.

System Architecture

  • Offline engine, frozen results: pipeline.py computes the incrementality estimate and recovery check once and writes results.json; the page renders those proven numbers so they never drift.
  • Governed live queries: a slim redemption table (observable columns only) is loaded into sql.js in the browser, and the four drill-down tracks run fixed, parameterised SQL over it.
  • Truth firewall: the hidden incrementality column lives only in an evaluation file that is never deployed, so the live dataset physically cannot reach it.
  • Keyless and free: the entire app runs client-side on static hosting, with no server and no API key.

# No build step and no server. Commit the folder; GitHub Pages serves it. # Live at: # https://lyven81.github.io/ai-project/projects/voucher-analyzer/demo.html

Production Notes

  • Fully static: no backend, no API key, and no per-visitor cost.
  • The in-browser SQL engine (sql.js) loads from a CDN; the ten core questions work even without it.
  • The hidden ground-truth file is excluded from the deployment by design, which keeps the "estimate the unobservable" claim honest.
