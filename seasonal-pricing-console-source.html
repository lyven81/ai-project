🏷️ Seasonal Pricing Console — Source

How the data, the estimator, the proof, and the governed query layer fit together

📁 Project Structure

projects/seasonal-pricing-console/ ├── demo.html # the console: self-contained, data embedded, sql.js in-browser ├── generate_data.py # synthetic generator with a baked-in elasticity ground truth ├── analyze.py # DiD elasticity estimator + recovery check + optimal price ├── build_db.py # builds the small SQLite DB the governed queries run on ├── results.json # the locked results the console renders └── README.md # how it works and how to run it

The build follows one order: data → brain → prove the brain → UI. Nothing is rendered until the estimate has recovered a known answer.

1. Data: a known truth baked in

The generator simulates three years of listing-nights and bakes a known price elasticity into each listing and season, then makes occupancy actually respond to price through that elasticity. Because the truth is known, the estimate can later be graded against it.

# generate_data.py (excerpt) TYPE_ELAST = {"Studio":-1.60,"1BR":-1.25,"2BR":-1.05,"3BR":-0.85} SEASON_ELAST_FACTOR = {"peak":0.80,"shoulder":1.00,"trough":1.20} # occupancy responds to price via the baked-in elasticity: price_response = (price/base) ** el # el = true elasticity occ = BASE_OCC * dw_term * DEMAND_SEASON[s] * price_response booked = int(rng.random() < occ)

2. Brain: difference-in-differences elasticity

Elasticity is the slope of log(occupancy) on log(price). The trap is that the host raises prices in peak season exactly when demand is high. Listing fixed effects and a weekend control remove that, so the price effect is identified from month-to-month price changes, with confidence intervals from the regression.

# analyze.py (excerpt) # y = ln(occupancy); regress on ln(price) with listing fixed effects + weekend X = np.column_stack([FE, np.log(price), is_weekend]) # FE = listing dummies beta, cov = wls(X, y, w) elasticity = beta[ei] # coef on ln(price) ci_low, ci_high = elasticity - 1.96*se, elasticity + 1.96*se

3. Prove: the recovery check (the hard gate)

Before any rate is trusted, the estimate is compared with the baked-in truth, and against a naive price-vs-occupancy read. The controlled estimator is 3.2x closer to the truth and lands inside the 95% interval in every cell.

# analyze.py (excerpt) recovery = dict( mean_abs_error = 0.224, # controlled estimator vs known truth naive_mean_abs_error= 0.713, # naive read (no controls), badly biased truth_in_ci_rate = 1.0, # truth inside 95% CI in 12/12 cells ) # optimal price: profit-max rate from the fitted curve, net of cost, # bounded to prices already observed that season (no extrapolation)

4. Governed query layer (in the browser)

The "Pricing Questions" tab answers four questions with fixed, parameterized read-only SQL run client-side via sql.js. The layer picks an approved track and binds values; it can never author its own SQL, and the hidden true sensitivity is never queried.

// demo.html (excerpt) — sql.js builds a small table, runs FIXED queries db.run("CREATE TABLE rates(listing_id, listing_title, property_type, season, elasticity, current_avg_price, optimal_price, ...);"); RATE_ROWS.forEach(r => st.run(r)); // fixed, read-only query — only values flow, never SQL const sql = "SELECT listing_title, ROUND(SUM(uplift_rm)) AS uplift " + "FROM rates GROUP BY listing_title ORDER BY uplift DESC"; const rows = run(sql); // rendered live in the browser
