How it works

The agent never touches the database. A governed tool layer sits in between — exactly the enterprise MCP Toolbox for Databases pattern, right-sized to a single-file SQLite store.

user question │ ▼ agent (Gemini Flash · or a deterministic router) ─ reasons WHICH tool to call; never writes SQL │ ▼ governed toolset (tools.yaml) ─ fixed queries, bound parameter values only │ ▼ SQLite (read-only) ─ right-sized for a one-desk clinic

The no-free-SQL guarantee

The agent reasons freely about which approved action to take — but can only ever run pre-written, parameterized queries against a read-only connection. Enforced at four layers:

1
The toolset is the agent's entire universe. No execute_sql tool is defined, so running arbitrary SQL is not a capability it has.
2
The SQL is fixed; only values flow. Each query lives in config; the agent supplies bound values only — it cannot add a WHERE, change a LIMIT, or inject a UNION.
3
Toolset scoping = access control. The front desk loads only its own tools; other roles would load different toolsets.
4
Read-only database connection (backstop). No INSERT/UPDATE/DELETE is physically possible — even if every layer above were bypassed.

execute_sql, list_tables and NL2SQL are all available in MCP Toolbox and deliberately switched off here. Choosing not to let the agent generate SQL is the whole point.

Bring your own key

Want the live, conversational version? Use your own free key.

The demo above is fully live and free with no key at all — the database and every guardrail run in your browser. To add Gemini's natural-language reasoning and summarising on top, the project uses a bring-your-own-key setup: you supply your own Gemini key, so nothing is ever hard-coded and the site owner pays nothing.

  1. Get a free Gemini key at aistudio.google.com/apikey.
  2. The key stays in your browser only (or a local .env file) — it is never sent to any server here and never committed to the repo.
  3. Your browser calls Gemini directly with your key. Your usage, your (tiny) cost — cents at this scale on the Flash tier.
  4. The data queries and the four guardrails are unchanged — the model still can't author SQL, dump records, or write anything.

Why BYOK? It's the only honest way to ship a live LLM demo on free static hosting: a shared key would be exposed in the page and abused. With BYOK the data layer stays free and local, and the reasoning layer is covered by whoever's actually using it.
