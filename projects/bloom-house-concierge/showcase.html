About this code showcase

Lucy is a customer-service agent for an online florist, built on the principle that an agent should be defined by what it cannot do. She talks to the customer directly and reaches exactly one of three outcomes on every message: resolve it with an approved tool, escalate it to a human, or politely decline it.

The whole demo runs client-side in one static page. The catalogue and orders live in a real SQLite database compiled to WebAssembly (sql.js) and queried in the browser. In a paying build the rule-based router is one Gemini call at the same boundaries; the gate, the tools, and the handoff stay exactly as they are here.

1 · The three-way gate — resolve, escalate, or decline

Before any answer, every message is sorted into one of three outcomes. Order matters: a decision-type request goes to a human before any tool runs, and an out-of-bounds request is declined before it can be mistaken for a handoff.

app.js — decide()
// 1. Decision-type (refund, cancel, change, complaint, upset) -> a human. const esc = escalationReason(text); if (esc) return { action: 'handoff', reason: esc.reason, situation: esc.situation }; // 2. Out of bounds (staff, internal, supplier, non-business) -> decline, never transfer. const dec = declineReason(text); if (dec) return { action: 'decline', reason: dec }; // 3. Information-type -> pick exactly one approved tool, or hand off if none fits. const tool = pickTool(text); if (!tool) return { action: 'handoff', reason: 'no approved tool matches' }; return { action: 'resolve', tool, reply: run(tool, text) };
Resolve → approved tool Escalate → Ms. Young Decline → politely refuse

2 · The governed toolset — Lucy's entire universe

Lucy can call exactly eight fixed tools and supplies parameter values only. She cannot author SQL, browse the schema, or invoke anything that writes. There is deliberately no processRefund, cancelOrder, or changeAddress tool, so those actions cannot happen autonomously, by design rather than by luck.

app.js — recommend_bouquets() runs a FIXED query against SQLite
recommend_bouquets({ occasion, maxBudget, colour }){ // The SQL string is fixed. Only bound values flow in. No injected WHERE/UNION. const FQ = "SELECT id,name,flower,colour,price,url FROM products " + "WHERE occasions LIKE '%,'||?||',%' AND price <= ? " + "AND (?='' OR colour=?) ORDER BY price DESC LIMIT 2"; let rows = runRows(FQ, [occasion, maxBudget||99999, colour||'', colour||'']); // the reason is what makes it a florist, not a search box return rows.map(p => ({ ...p, reason: FLOWER_MEANING[p.flower] })); }

Eight approved tools: recommend_bouquets, get_delivery_cutoff, get_coverage_areas, get_care_tips, get_hours, get_store_location, flower_meaning, get_order_status (read-only).

3 · The escalation gate — decision-type goes to a human

Capability is easy; knowing when to stop is the product. Anything that commits money, changes an order, or carries emotion has no tool, so it routes to the founder, surfaced to the customer as the manager, Ms. Young.

app.js — escalationReason()
function escalationReason(t){ if (matchesAny(t, COMPLAINT)) return { reason: 'complaint or failed delivery' }; if (matchesAny(t, MONEY)) return { reason: 'touches money (refund / payment)' }; if (matchesAny(t, ORDER_CHG)) return { reason: 'asks to change or cancel an order' }; if (matchesAny(t, CUSTOM_BULK)) return { reason: 'custom, wedding or bulk request' }; if (matchesAny(t, UPSET)) return { reason: 'customer sounds upset' }; return null; }

4 · The decline rule — private and off-topic, refused not transferred

A real shop assistant should not leak staff details or answer questions that have nothing to do with an order. These are refused with a fixed line and are never sent to a person. Ms. Young is a publicly designated contact, so naming her is fine; her schedule and personal details are not.

app.js — the nine declined categories
const DEC = { staffCount, staffSchedule, staffContact, staffName, supplier, margin, ownership, internal, nonbiz }; // "How many staff?", "Is Ms. Priscilla in today?", "Her personal number?", // "Who supplies you?", "Your margin?", "Who owns this?", "Weather in KL?" // -> "I'm sorry, but I'm unable to help with that." (no transfer)

5 · Warm handoff — the human never starts cold

When the gate escalates, Lucy writes a short summary so the customer does not repeat themselves. The summary is shown only on the owner side, so the customer feels a person stepping in, not a bot logging a ticket.

Data model & key decisions

A real SQLite database (products, orders) built by build_db.py and embedded in the page as base64, so the same fixed queries run live on GitHub Pages and from file://.

  • Defined by its limits: three outcomes, eight tools, no write path. The agent reads; it never commits money or changes an order.
  • Real database, not a mock: recommendations and order lookups are genuine SQL SELECTs against SQLite, not hard-coded replies.
  • Grounded recommendations only: Lucy suggests products that exist in the catalogue, with a flower-meaning reason. No invented bouquet, no invented price.
  • Decision before tool, decline before handoff: the order of checks is the safety model.
  • Seamless to the customer: the handoff summary and decision panel are owner-facing; the customer never sees the machinery.
