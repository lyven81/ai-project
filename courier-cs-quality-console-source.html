📦 Courier CS Quality Console — Source

How the measurement is built, proven, and governed

📁 Project Structure

The build runs foundation-up: data with a known truth, then the scoring brain, then a recovery check that proves it, then the console and the governed assistant on top.

projects/courier-cs-quality-console/ ├── generate_log.py # synthetic courier CS log + ground-truth key ├── analyze.py # the measurement brain -> results.json ├── recover.py # recovery check (Wilson CIs) -> recovery.json ├── build_console.py # assembles the console from results.json ├── build_assistant.py # the standalone governed assistant ├── _console_template.html # the 5-tab console shell ├── _assistant_template.html ├── ground-truth-key.json # the answer key (correct outcome + checklist per reason) ├── results.json # the measured metrics (single source for the UI) ├── recovery.json # the recovery-check result ├── demo.html # the built console (the live demo) └── data/ ├── courier-cs-log.csv # deployed log (no truth columns) ├── courier-cs-eval.csv # same rows + the baked-in truth └── courier-cs-scored.csv # the brain's derived columns

1. The ground-truth answer key

Every reason carries the correct outcome and a "good reply must include" checklist. This is the known truth the whole build is scored against.

{ "id": 2, "reason": "Tracking says delivered but not received", "correct_outcome": "escalate", "handled_by": "human", "good_response_must_include": [ "apologize for the situation", "confirm the delivery record: time, location, proof of delivery", "open a proof-of-delivery investigation with a reference number", "advise checking household / neighbour / guardhouse", "set a callback SLA (for example 24 to 48 hours)" ], "bad_response_example": "Our system shows it was delivered, please check around your house." }

2. The measurement brain

It reads only the deployed log (no truth), classifies the reason from the message text, then scores the agent's gate decision against the answer key. It estimates the verdict, it does not read it.

def classify(msg): # keyword-signature reason classifier m = msg.lower(); best, score = 0, 0 for rid, kws in SIG.items(): s = sum(1 for k in kws if k in m) if s > score: best, score = rid, s return best for r in log: rid = classify(r["customer_message"]) correct_oc = REASONS[rid]["correct_outcome"] oc_correct = (r["agent_outcome"] == correct_oc) # graded against the key quality = score_response(r["agent_response"], rid) # checklist coverage

3. The recovery check (the credibility centrepiece)

Before any number is shown, the brain's derived values are graded against the held-out truth, with Wilson 95% confidence intervals, and against a naive baseline.

def wilson(k, n, z=1.96): # 95% confidence interval for a proportion p = k/n; d = 1 + z*z/n c = (p + z*z/(2*n))/d h = (z*math.sqrt(p*(1-p)/n + z*z/(4*n*n)))/d return (round(100*max(0,c-h),1), round(100*min(1,c+h),1)) # result: reason recovery 100% (CI 99.7-100, vs 10.5% naive), # verdict agreement 100%, 0 disagreements with the truth.

4. The governed assistant (Klinik fixed-query discipline)

The "Ask the data" tool maps a question to one approved query and returns the exact figure. Anything off the menu is refused; raw records are never shown.

function ask(){ const t = input.value.toLowerCase().trim(); for (const a of APPROVED) // approved queries only if (a.kw.some(k => t.includes(k))) return showAns(a.q, a.a); for (const k in SNIPS) // governed redacted examples if (SNIPS[k] && t.includes(k)) return showSnip(k); return refuse(); // off-menu -> polite refusal }

The console itself is assembled by build_console.py, which injects results.json into the template, so the screen can never disagree with the validated pipeline.
