📋 Project Overview & Problem Statement

Challenge: Short-term rental hosts price their listings by gut feel, usually with one flat seasonal rule. That rule misreads how differently each property responds to price: small studios sit overpriced and run empty nights, while larger apartments sit underpriced and leave money on the table. Hosts have no trustworthy way to see how occupancy responds to price, season by season.

Solution: The Seasonal Pricing Console takes a host's booking history and shows, in plain language, how price-sensitive each listing is by season, the nightly rate that earns the most for each property in each month, and a recovery-check proof that the numbers can be trusted. It runs entirely in the browser, with a governed query layer that answers pricing questions live from the data without ever writing free-form SQL.

Key Benefits

  • Season-aware pricing: Each month is classified into peak, shoulder, or value, and every rate follows from it
  • Measured price sensitivity: See how many bookings you lose for a 10% price rise, by room type and season
  • A ready price sheet: The recommended nightly rate for all 15 listings across 12 months, in one table
  • Trust before you act: The estimate recovers a known answer 3.2x better than the naive read, inside the 95% interval every time
  • Governed, not guessed: Live questions run as fixed read-only queries that never see the hidden true sensitivity

🖥️ Application Features

📅 Overview & Season Calendar

Headline KPIs (revenue, occupancy, projected uplift) plus the month-to-season calendar and the recovery-check proof, so the host sees what season each month is in and why the numbers can be trusted.

📉 Price Sensitivity

How many bookings out of 100 each room type loses for a 10% price rise, by season. Studios are most sensitive, larger units least, which is why studios are cut and larger units raised.

🏷️ Recommended Rates

A full price sheet: the suggested nightly rate for every listing in every month, colour-coded by season, staying within prices the host has already tested.

💬 The Pricing Questions

The ten questions a host asks most, answered from the locked results, plus four live read-only SQL queries that run in the browser over the listing-season rates.

🤖 Analytics Engine & Governed Intelligence

🔒 Governed Query Layer

The live questions run as fixed, parameterized read-only SQL via sql.js. The layer can only pick an approved track and bind values; it can never author its own SQL, list tables, or dump records. It never sees the hidden true price sensitivity.

📐 Difference-in-Differences Elasticity

Price sensitivity is estimated with listing fixed effects and a weekend control, identified from the host's month-to-month price changes, with 95% confidence intervals. This is rigor a chatbot hand-waves.

✅ Recovery-Check Eval

Before any rate is shown, the estimate is checked against a baked-in known elasticity: mean error 0.224 vs 0.713 for a naive read, inside the 95% interval in 100% of the 12 type-season cells.

💰 Season-Aware Optimal Pricing

From the fitted demand curve and the cost of servicing a booked night, the engine derives the profit-maximizing nightly rate per listing per season, bounded to prices already observed.

🛠️ Technical Architecture & Implementation

Frontend Stack

NumPy pandas Synthetic generator (baked-in truth) DiD estimator Recovery-check eval

Deployment

GitHub Pages (static) No backend No API key

System Architecture

  • Data -> Brain -> Prove -> UI: a synthetic generator bakes in a known elasticity, the estimator recovers it blind, the recovery check gates it, and only then does the console render
  • Self-contained page: the locked results are embedded inline; the console runs by opening the file, offline-capable
  • In-browser SQLite: sql.js builds a small rates table from the listing-season data and runs four fixed read-only queries client-side
  • Governed toolset: the query layer binds parameter values only, never composes SQL, and the eval-only truth column is never queried

📖 Development Setup & Installation Guide

Prerequisites

  • A browser to run the console (no install needed)
  • Python 3 with NumPy and pandas only if regenerating the analysis

Quick Start

# Clone the repository git clone https://github.com/lyven81/ai-project.git cd ai-project/projects/seasonal-pricing-console # Open the console (no build, no server) open demo.html # Regenerate the analysis (optional) python generate_data.py # synthetic data with baked-in elasticity python analyze.py # DiD estimator + recovery check -> results.json python build_db.py # build the SQLite DB

🚀 Deployment

# Static deployment on GitHub Pages, no server or key # The console is a single self-contained HTML file at: # projects/seasonal-pricing-console/demo.html # Push to the repo and it is live on GitHub Pages.

Production Notes

  • The console is fully static: data is embedded and sql.js loads from a CDN, so there is nothing to host beyond the file
  • No API key, no backend, no running cost
  • The data is synthetic; the same pipeline runs unchanged on a host's real booking export
