📋 Project Overview & Problem Statement

Challenge: A courier runs an AI chatbot alongside a human support team, across website chat, WhatsApp and the app. Leadership cannot tell how well that mix is performing, or where it is quietly losing customers. Pasting a few chats into ChatGPT gives an opinion on one conversation; it cannot measure thousands, grade them consistently, or prove the numbers.

Solution: A management layer that reads the whole conversation log, scores every agent decision against an agreed answer key, and turns it into a plain-language monthly review: an overall grade, the trend on the signals that matter, the single biggest issue, per-agent appraisals, a coaching playbook for training recruits, and a prioritised action plan. The technicalities stay under the hood.

Key Benefits

  • Answer first: a five-second glance gives the grade, the trend, and the one thing most worth fixing.
  • Catches what a chatbot and a busy human cannot: high-stakes cases (claims, disputes, complaints) being closed instead of escalated, across every conversation.
  • Fair, individual appraisals: a ready scorecard for each agent, with real examples from their own conversations.
  • Trains new recruits: a coaching playbook of what a good reply looks like for every contact reason.
  • Trustworthy: the measurement is graded against a known answer key, recovering the true contact reason on 100% of conversations before any number is used.

🖥️ Application Features

Overview

The verdict first: an overall grade, three signal tiles with the direction versus last month, and the single biggest issue with a one-tap jump to the fix. A governed "Ask the data" box sits underneath.

Team this month

The detail behind the signals: what drives customer happiness, what customers contacted about, and where service is leaking.

Agent review

A per-agent appraisal scorecard for the AI assistant and each human agent: grade, strengths, where to coach, real example conversations, and talking points for the meeting.

Coaching playbook

The right move and a strong-reply checklist for every one of the 20 contact reasons, plus a weak reply to avoid. Built for training recruits.

Action plan

Three changes in order of payoff, each naming who it affects and what to expect at the next review.

Ask the data (governed)

A fixed menu of approved questions over the figures, with a free-text box. Returns the exact number, never raw records, refuses anything off the menu, and returns redacted example conversations on request.

🤖 AI Integration & Intelligence

Measurement brain

Reads each conversation, classifies its contact reason, scores the agent's gate decision (resolve / escalate / decline) against the answer key, and grades the reply against a checklist. It scores decisions, it does not just display them.

Recovery check

Before any number is shown, the brain is graded against a held-out truth: 100% reason recovery and an exact verdict match, with Wilson confidence intervals, beating a naive "trust the agents" baseline that would report zero problems.

Governed assistant

The "Ask the data" tool follows a fixed-query discipline: approved questions only, exact counted answers, no free database queries, no record dumps, and a polite refusal for anything off the menu.

Privacy by design

The conversation log holds customer details, so the assistant never shows raw records. Its one "show me examples" tool returns a few conversations with names and numbers redacted.

System Architecture (foundation-up)

  • Data layer: a conversation log with a known ground-truth answer key baked in, so the method can be validated.
  • Brain: classifies the reason, scores the gate decision and the reply, and aggregates the metrics into results.json.
  • Prove it: a recovery check grades the brain against the held-out truth before anything is built on top.
  • Surface: the console reads results.json at build time, so the screen can never drift from the validated numbers.
  • Assistant: the only piece that needs a live key in a paid build; static and offline here.

📖 Development Setup & Build Guide

Prerequisites

  • Python 3 (standard library only for the pipeline)

Build the console from scratch

# Clone the repository git clone https://github.com/lyven81/ai-project.git cd ai-project/projects/courier-cs-quality-console # Run the pipeline (data -> brain -> prove -> build) python generate_log.py # synthetic conversation log + ground-truth key python analyze.py # the measurement brain -> results.json python recover.py # the recovery check -> recovery.json python build_console.py # writes the self-contained console python build_assistant.py # writes the standalone governed assistant # Open the result # demo.html (the console) runs by double-click, no server needed

Files

  • generate_log.py — synthetic courier CS log with the truth baked in
  • analyze.py — the measurement brain
  • recover.py — the recovery check (Wilson CIs)
  • build_console.py — assembles the console from results.json
  • ground-truth-key.json — the answer key (correct outcome + good-reply checklist per reason)

🚀 Deployment

# Static (this demo): no server, no key # The console is fully self-contained and runs on GitHub Pages or by double-click. # Production track (a paying client): # - governed assistant runs the same fixed tools live via sql.js / MCP Toolbox # - over a real database (SQLite + Litestream, or Cloud SQL), deployed on Cloud Run

Production Notes

  • The portfolio version is static and synthetic; real client logs replace the data file unchanged downstream.
  • On real logs, the recovery check is run against a hand-labeled sample as the standing quality gate.
  • Trigger-based escalation (escalate only when tracking is stale or an attempt failed) is a documented production enhancement needing tracking-timestamp fields.
