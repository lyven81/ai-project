OSINT intelligence aggregator with 27 data sources, 3D globe visualization, and multi-LLM briefings

Node.js 22+ Express 5 Multi-LLM Docker Telegram Bot Discord Bot

About This Code Showcase

This curated selection shows how Crucix pulls data from 27 OSINT feeds in parallel, manages a delta/memory engine to detect significant changes, and serves everything through a Jarvis-style dashboard with SSE live updates, Telegram bot commands, and Discord alerts.

The architecture is modular: each data source is a standalone fetcher under apis/sources/, the delta engine tracks change significance, and multiple LLM providers can generate intelligence briefings.

Core: Intelligence Engine & Live Updates

The server initializes a delta/memory engine, multi-LLM provider, and alert bots at startup. It runs sweep cycles every 15 minutes (configurable), fetching all 27 OSINT sources in parallel, detecting significant changes via the delta engine, and pushing live updates to connected dashboard clients via Server-Sent Events.

Configuration & Docker Deployment

All settings support environment variable overrides — LLM provider, Telegram/Discord credentials, delta engine thresholds, and refresh interval. The Docker setup uses Alpine Linux with a built-in health check endpoint.

crucix.config.mjs — Full Configuration
export default { port: parseInt(process.env.PORT) || 3117, refreshIntervalMinutes: parseInt(process.env.REFRESH_INTERVAL_MINUTES) || 15, llm: { provider: process.env.LLM_PROVIDER || null, // anthropic | openai | gemini | codex apiKey: process.env.LLM_API_KEY || null, model: process.env.LLM_MODEL || null, }, telegram: { botToken: process.env.TELEGRAM_BOT_TOKEN || null, chatId: process.env.TELEGRAM_CHAT_ID || null, channels: process.env.TELEGRAM_CHANNELS || null, }, discord: { botToken: process.env.DISCORD_BOT_TOKEN || null, channelId: process.env.DISCORD_CHANNEL_ID || null, webhookUrl: process.env.DISCORD_WEBHOOK_URL || null, }, delta: { thresholds: { numeric: {}, count: {} } // Override sensitivity per source } }

Technical Implementation Notes

Key Design Decisions

  • 27 parallel source fetchers: Each OSINT source (satellite fire detection, flight tracking, radiation monitoring, market prices, conflict data, etc.) has its own module under apis/sources/, all fetched concurrently during each sweep
  • Delta/memory engine: A three-tier memory system (hot/warm/cold) tracks changes over time, detecting when a data point moves significantly enough to warrant an alert
  • Multi-LLM support: The LLM layer supports Anthropic, OpenAI, Gemini, and Codex — switchable via a single environment variable
  • Server-Sent Events (SSE): Dashboard clients receive live push updates when new sweep data arrives, no polling required
  • Two-way bot commands: Telegram users can query system status (/status) or trigger manual sweeps (/sweep) directly from chat
  • Single production dependency: Only Express is required — all 27 source fetchers use Node.js built-in fetch()
