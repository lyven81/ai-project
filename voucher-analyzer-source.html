📊 Voucher Analyzer

Incrementality engine, recovery check, and governed query layer

🔍 About This Code Showcase

These three excerpts are the heart of the project: an incrementality estimate that runs blind to the answer, the recovery check that grades it against a hidden truth, and the governed query layer that powers the live questions.

Dataset-generation plumbing is omitted for clarity. The key idea throughout: the per-order ground truth (was_incremental) is withheld from everything the page can see, so the tool genuinely estimates rather than reads the answer.

🧮 The Analytical Brain: Incrementality, Estimated Blind

The core skill is separating the sales a voucher truly caused from buyers who would have purchased anyway. The estimate uses a baseline-counterfactual (difference-in-differences style): learn each category's ordinary order rate from non-campaign days, then treat the excess during campaigns as incremental. It never looks at the truth column.

📄 pipeline.py: withhold the truth, then estimate
# DEPLOYED dataset: the per-order truth column is WITHHELD. # The assistant must never see was_incremental. df.drop(columns=["was_incremental"]).to_csv("voucher_orders.csv", index=False) # EVAL copy WITH the truth, for the recovery check only. Never deployed. df.to_csv("voucher_orders_eval.csv", index=False) # Incrementality ESTIMATE via baseline-counterfactual (DiD style), per category. # Blind to truth: compare in-campaign volume to the baseline learned off-campaign. est_share = {} for c in CATS: base_rate = ((df.category==c) & (~df.in_window) & (df.voucher_id=="none")).sum() / max(nonwin_days, 1) win_orders = ((df.category==c) & (df.in_window)).sum() exp_base = base_rate * len(win_days) incr_vol = max(0.0, win_orders - exp_base) # orders the campaign actually added redemptions = (v.category==c).sum() est_share[c] = float(np.clip(incr_vol / max(redemptions, 1), 0, 1))

✅ The Proof: Recovery Check

A blind estimate is only worth trusting if it lands on the real answer. The recovery check compares the per-category estimate against the baked-in truth and reports the error. It recovers an overall 45 percent incremental share against a true 46 percent, while the naive "every redemption is new" read would claim 100 percent.

📄 pipeline.py: grade the estimate against the hidden truth
# PHASE 3: recovery check, only here is the truth column read. true_share = {c: float(v[v.category==c]["was_incremental"].mean()) for c in CATS} mae = np.mean([abs(est_share[c] - true_share[c]) for c in CATS]) recovery = dict( by_category = {c: dict(estimate=round(est_share[c],3), truth=round(true_share[c],3)) for c in CATS}, mean_abs_error = round(mae, 3), # about 0.085 naive_share = 1.0, # the naive read overstates true_overall = round(v.was_incremental.mean(), 3), # 0.46 est_overall = round(np.mean([est_share[c] for c in v.category]), 3), # 0.446 )

🔐 The Governed Query Layer (client-side, keyless)

The live questions in the app run real SQL in the browser via sql.js, on the Klinik Dr Fang pattern. Each track is a fixed, parameterised, read-only statement. The model (or the click handler) can pick a track and bind a value; it can never write, extend, or alter the SQL, and the deployed table has no truth column to reach.

📄 demo.html: fixed, parameterised, read-only tools
// Values only (e.g. {':v': 'F1'}); the SQL itself is fixed. function run(sql, params) { const st = db.prepare(sql); if (params) st.bind(params); // bind a value, never author SQL const out = []; while (st.step()) out.push(st.getAsObject()); st.free(); return out; } // Track 13: do vouchers bring new buyers or reward returning ones? const rows = run( "SELECT customer_type, COUNT(*) AS orders " + "FROM orders WHERE voucher_id = :v GROUP BY customer_type", {':v': 'F1'} );

⚙️ Why This Design

  • Offline engine, frozen results: the brain runs once in Python and writes results.json; the page renders proven numbers, so the displayed answers never drift.
  • Truth firewall: was_incremental exists only in the eval file, which is never deployed; the live data physically cannot surface the answer key.
  • Match the tool to the job: incrementality is a counterfactual computation, so it stays in Python; the observable drill-downs are simple aggregations, so they run as governed SQL in the browser.
  • Keyless and free: the whole app is static, with no backend and no API key, so it runs on GitHub Pages at zero cost.
