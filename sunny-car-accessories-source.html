What this is

Sunny Car Accessories is a working multi-category e-commerce storefront — 18 sneakers… sorry, 18 car-accessory SKUs across Interior / Exterior / Performance / Tech — that exposes structured tools to AI agents via the proposed WebMCP W3C web standard (navigator.modelContext).

Humans see a normal storefront. Agents see a typed contract: "Here are my 7 tools, here are their parameters, here's how to call them." No screenshotting. No pixel-clicking. No abandoned agent carts.

Why this matters now: WebMCP shipped stable in Chrome 146 (April 2026). It's the first time websites can natively expose structured tools to AI agents in the browser. For Malaysian SMEs, this is the agent-friendly storefront layer that turns "we have a website" into "any AI agent can shop here." First-mover positioning on a real W3C standard, not vendor-locked.

Try It

The demo is a single static HTML file — no backend, no hosted services, no API keys. Open it, browse like a human, then open the WebMCP Inspector Panel (bottom-right) to see what agents see.

Two paths to test

  • Path 1 — Shop as a human. Click categories, add to cart, check out. Standard shop UX. The Inspector Panel sits unobtrusive in the corner.
  • Path 2 — Try as an agent. Open the Inspector → "Try as Agent" tab → click any "▶ Run [tool]" button. The page reacts to typed function calls, not pixel clicks. Watch the Log tab record every invocation with parameters + structured JSON results.

What this proves: per-view tool scoping (the killer WebMCP feature) works as designed. Agents on the cart page see cart-management tools. Agents on the checkout page see checkout with its mandatory human-in-the-loop gate. This isn't simulated — it's provideContext() being called fresh on each view, exactly per the W3C spec.
Browser-support disclaimer. WebMCP is in W3C incubation as of April 2026. The navigator.modelContext API is currently flag-gated even on Chrome 147 stable — the deck's "146+ stable" claim is aspirational; live API access requires chrome://flags/#enable-experimental-web-platform-features or a similar flag enabled.

On any browser without the API, the Inspector Panel runs in compatibility mode — it still demonstrates exactly what tools an agent would see, the per-view scoping, the JSON schemas, and the confirmation flow. The only thing that doesn't work in compat mode is the link to a real external agent (which requires the browser to expose the API). The same code switches automatically to live agent integration the moment the browser enables WebMCP — no code change needed on this page.

To see the green ✅ today: Chrome 147 → chrome://flags → search "model context" or enable Experimental Web Platform Features → relaunch → reload this page.

Why It Matters — 5 Business Values for Merchants

Why a Malaysian SME should add WebMCP to their existing storefront:

1. Get found by AI shopping agents

Customers are increasingly asking ChatGPT, Gemini, Claude, and Auto Browser to do the shopping — not just search. Agents pick websites they can reliably interact with. WebMCP-enabled is to AI agents what mobile-friendly was to Google in 2015. First-mover visibility on agent-driven traffic.

2. Higher checkout conversion from agent traffic

Without WebMCP, agents shop by screenshotting and guessing — abandoned carts, wrong variants added, price drift. With WebMCP, the agent calls addToCart({sku, variant, quantity}) directly. Recovered checkouts. No "I ordered red, got blue" disputes.

3. 24/7 customer support deflection

Tools like getStockLevel, checkFitment, getReturnPolicy let agents answer pre-sale questions without the customer pinging WhatsApp / FB Messenger. Hours of staff time saved on FAQ-tier queries — especially valuable for solo SMEs where the owner is the support desk.

4. A "public API" at frontend-developer cost

Most SMEs would like to expose APIs but can't justify the cost (gateway, OAuth, docs, versioning). WebMCP gives 80% of the API value at 10% of the cost. A frontend dev adds tool registrations to existing pages — no new servers, no auth tokens to issue.

5. Measurable agent-traffic analytics

submitEvent.agentInvoked flags when an agent (not a human) took an action. For the first time, sites can split traffic into human vs agent and measure separately. Board-level question "how much revenue came from AI-mediated purchases?" gets a real answer.

How It Works

The paradigm shift, in one comparison

Before — Inference-based pixel-clicking

  • Parse raw HTML / take screenshots
  • LLM guesses which element is "Add to Cart"
  • Simulate mouse click at (x, y)
  • Hope the DOM hasn't changed
  • Retry when CSS update breaks everything
  • 2,000–5,000+ tokens per page

After — Contract-based WebMCP

  • Website registers tools explicitly
  • Agent discovers tools + JSON schemas
  • Direct call: addToCart({sku, variant, qty})
  • Structured JSON response returned
  • UI changes don't break the contract
  • 20–100 tokens per call (89% reduction)

Per-view tool scoping (WebMCP's killer feature)

Tool surface morphs as the user (or agent) navigates. On the homepage, only searchAccessories is registered. Walk to a product page — three more tools appear (and search disappears since it's not the action context). Hit checkout — destructive tools (checkout) appear with mandatory human confirmation; cart-mutation tools disappear because they don't belong in the finalize step.

This is what slide 20 of the WebMCP deck means by "tools naturally scoped per page (3–5, not 43)." Agents see 3–5 contextual tools at a time, not a monolithic API surface. Lower token cost, better tool selection, security boundary built in.

// Implemented via:
function updateWebMCPContext() {
  const tools = VIEW_TOOLS[state.currentView]
    .map(name => TOOLS[name]);
  navigator.modelContext.provideContext({ tools });
}
// Called on every view transition.

Architecture extension — single-category vs multi-category

This demo is multi-category (Interior, Exterior, Performance, Tech). The same WebMCP pattern scales to single-category niches (sneakers, watches, perfumes) — fewer schemas, smaller tool surface. Multi-category is the harder case because each category may have its own typed schema; if the pattern works here, it works for single-category too. Reference: webmcp-knowledge-base.md §19.
